## 2026-07-19 update — re-baseline + "smarter model" eval (flash vs. pro)

The prompt/protocol changed since the baseline above (JEE physics pack:
orbit/spring/wave/ray, axes grid toggle, wish op, steps card), so the
`deepseek-v4-flash` numbers were re-measured on the current prompt before
comparing against `deepseek-v4-pro`. Both are live 10/10-case runs from
`server/`, same command shape, only `BOARD_MODEL` differs
(`BOARD_MODEL=deepseek-v4-pro` prefix overrides `.env` — verified with
`node --env-file` that shell env wins over `--env-file`, so no `.env` edit
was needed for the run itself).

- **Date:** 2026-07-19
- **Flash run:** `generatedAt` `2026-07-19T12:58:08.663Z`
- **Pro run:** `generatedAt` `2026-07-19T12:58:53.554Z`

### Comparison table

| Metric | Target | flash (re-baseline) | pro | Δ (pro − flash) |
|---|---|---|---|---|
| Validity % | ≥ 95 | 98.7 | **100.0** | +1.3 |
| Semantic pass % | ≥ 80 | **96.6** | 89.7 | **−6.9** |

All four numeric targets: **flash PASS / PASS / PASS / PASS**, **pro PASS /
PASS / PASS / PASS** — pro clears every target, but by a much thinner margin
on TTFA (1488ms vs. a 1500ms ceiling — a single slower case away from
failing) and it regresses semantic pass% (three cases newly fail a check —
`projectile-45` "mentions range", `ask-mcq-format` "answer matches an
option verbatim", plus the pre-existing `numberline-fractions` "mentions
fraction" miss — vs. flash's one pre-existing miss). Pro also emits fewer,
denser actions per turn (e.g. 3 visuals for `quadratic-intro` vs flash's 5),
consistent with a more deliberate/less chatty model.

### Recommendation

Per the decision rule (user asked for "a smarter model," measured, and pro
clears all 4 targets), **`BOARD_MODEL` is set to `deepseek-v4-pro`** in
`.env` (uncommitted, per repo convention). This is a real trade: pro is more
disciplined about board validity (100% vs 98.7%, zero sanitize drops) but is
measurably slower to first visual (1488ms median, within 12ms of the 1500ms
ceiling — the next infra hiccup or longer prompt will likely tip it over)
and less reliable on secondary semantic checks (89.7% vs 96.6%). If TTFA
regresses further in production, the fallback is trivial: unset
`BOARD_MODEL` overrides back to `deepseek-v4-flash` in `.env`, which stays
fully documented in `.env.example` as the fast/cheap option. Treat pro's
current numbers as barely-passing, not comfortably-passing — this is a
model to keep re-evaluating, not a settled choice.

## 2026-07-21 update — genuine OpenAI-API provider path (BOARD_PROVIDER=openai)

The server gained a second, real provider seam (`server/src/openai.ts` +
`server/src/provider.ts`): Chat Completions streaming against DeepSeek's
**OpenAI-compatible** endpoint (`https://api.deepseek.com`, NOT the
`/anthropic` compat surface the runs above used), forced function
`tool_choice` + `strict: true`, `stream_options: {include_usage: true}`. Same
key (`ANTHROPIC_AUTH_TOKEN`) works on both endpoints.

**Model-id finding (empirical, live):** `deepseek-v4-pro` / `deepseek-v4-flash`
/ `deepseek-reasoner` are all "thinking mode" on this endpoint and reject a
forced `tool_choice` outright — `400 Thinking mode does not support this
tool_choice` — with no equivalent of the Anthropic-compat surface's
`thinking: {type:'disabled'}` escape hatch exposed over Chat Completions.
**`deepseek-chat`** is the one model id on this endpoint that accepts forced
function-call `tool_choice`, so `BOARD_MODEL`/`PLANNER_MODEL` must be
`deepseek-chat` under `BOARD_PROVIDER=openai` — a different value than the
Anthropic-compat path's `deepseek-v4-pro`, not a drop-in swap.

**Live full-lesson run:** `BOARD_PROVIDER=openai BOARD_MODEL=deepseek-chat
PLANNER_MODEL=deepseek-chat tsx --env-file=../.env scripts/drive-lesson.ts

the existing disk cache; every board turn streamed live through
`streamBoardTurnOpenAI`), 14 turns (11 teach/check beats + 1 chat aside + 1

actions** across the entire run.
